# DecisionJury MVP 文档

## 1. 项目名称

DecisionJury：基于多 Agent 协作的日常冷静决策助手。

## 2. MVP 目标

在三周内完成一个可运行的 Web Demo。用户可以提交购物决策或时间决策，系统通过多 Agent 辩论、历史记录 RAG 检索和 MCP 工具调用，生成一份可解释的冷静决策判决书。

MVP 的核心不是“给建议”，而是完成一个 Agent 工作流：

```text
用户提交决策
  -> 多轮补全信息
  -> RAG 检索历史记录
  -> 调用 MCP 工具
  -> 正反方 Agent 辩论
  -> 法官 Agent 裁决
  -> 生成判决书和后续动作
```

## 3. 目标用户

### 3.1 购物决策用户

典型用户：

- 大学生。
- 每月预算有限。
- 容易被短视频、社交平台、促销活动种草。
- 经常买完后发现使用频率很低。

典型需求：

- 买之前有人帮忙分析是否真的需要。
- 想知道这次购买对预算影响多大。
- 希望系统能提醒自己过几天再决定。

### 3.2 时间决策用户

典型用户：

- 学生干部。
- 社团成员。
- 小组项目成员。
- 任务较多但不擅长拒绝的人。

典型需求：

- 判断一个活动或任务是否值得参加。
- 判断本周时间是否还能承受。
- 希望系统帮忙列出风险和替代安排。

## 4. MVP 范围

### 4.1 必做功能

- 创建决策案件。
- 支持两类案件：
  - 购物决策：买不买某个商品。
  - 时间决策：要不要参加活动、接任务或投入某件事。
- 多轮追问用户信息。
- 多 Agent 流程：
  - 输入解析 Agent。
  - 正方 Agent。
  - 反方 Agent。
  - 法官 Agent。
- RAG 检索：
  - 检索历史决策记录。
  - 检索后悔记录、闲置记录、预算偏好、时间偏好。
- 至少两个 MCP 工具：
  - 成本计算工具。
  - 冷静期提醒/观察清单工具。
- 多轮状态管理。
- 生成最终判决书。
- 前端展示完整流程。

### 4.2 可选功能

- 决策评分工具。
- 替代方案推荐。
- Docker 部署。
- RAG 质量评测指标。
- Agent 调用链路可视化。
- 2-5 分钟演示视频。

### 4.3 暂不做功能

- 医疗、法律、投资、借贷、就业离职、亲密关系等高风险决策。
- 真实支付。
- 真实下单。
- 真实电商账号登录。
- 全网实时比价。
- 真实个人日历深度集成。
- 手机 App。

## 5. 用户故事

### 5.1 购物决策

```text
作为一个预算有限的学生，
我希望在购买高价商品前获得理性分析，
以便避免因为冲动消费导致月底预算不足。
```

验收标准：

- 系统能询问价格、预算和购买动机。
- 系统能检索历史闲置或后悔记录。
- 系统能计算预算占比。
- 系统能给出购买、暂缓、不建议购买或寻找替代品的辅助建议。

### 5.2 时间决策

```text
作为一个经常被社团和课程任务占满的学生，
我希望在接受新任务前评估时间成本，
以便避免任务堆积和延期。
```

验收标准：

- 系统能询问占用时间和当前任务。
- 系统能检索历史拖延或任务冲突记录。
- 系统能计算时间压力。
- 系统能给出参加、部分参加、推迟或拒绝建议。

## 6. 功能优先级

| 优先级 | 功能 | 是否必须 | 说明 |
|---|---|---|---|
| P0 | 创建决策案件 | 是 | 没有案件就没有后续流程 |
| P0 | 多 Agent 辩论 | 是 | 项目核心亮点 |
| P0 | RAG 历史检索 | 是 | 课程硬性要求 |
| P0 | 成本计算 MCP 工具 | 是 | 课程硬性要求 |
| P0 | 冷静期提醒 MCP 工具 | 是 | 课程硬性要求 |
| P0 | 多轮状态管理 | 是 | 课程硬性要求 |
| P1 | 前端可视化展示 | 是 | 答辩演示需要 |
| P1 | 判决书生成 | 是 | 最终结果载体 |
| P1 | Docker 部署 | 建议做 | 加分项 |
| P2 | 决策评分工具 | 可选 | 时间不够可砍 |
| P2 | Agent 调用链路看板 | 可选 | 有助于答辩，但不是 MVP 必需 |

## 7. 数据范围

第一版不需要真实大规模数据，只需要足够支撑演示和测试。

### 7.1 历史购物记录样例

至少准备 10 条：

- 闲置物品记录。
- 后悔购买记录。
- 高价购买记录。
- 预算超支记录。
- 高频使用且值得购买记录。

示例：

```text
2026-06-05，购买机械键盘 399 元，实际使用频率较低，用户评价：冲动购买。
2026-06-12，购买学习台灯 129 元，使用频率高，用户评价：值得。
```

### 7.2 历史时间记录样例

至少准备 10 条：

- 因参加活动导致任务延期。
- 合理参加活动后获得收益。
- 拒绝任务后保证学习时间。
- 同时接多个任务导致压力过大。

示例：

```text
2026-06-10，参加社团活动占用 8 小时，导致课程作业延期。
2026-06-15，参加技术分享占用 2 小时，收获较高，值得参加。
```

## 8. 裁决类型

### 8.1 购物决策

- buy：建议购买。
- delay：建议暂缓。
- reject：不建议购买。
- alternative：建议寻找替代品。

### 8.2 时间决策

- accept：建议参加。
- partial_accept：建议部分参加。
- delay：建议推迟决定。
- reject：建议拒绝。

## 9. 判决书内容

最终判决书必须包含：

- 案件摘要。
- 用户动机。
- 正方观点。
- 反方观点。
- RAG 历史证据。
- MCP 工具计算结果。
- 最终裁决。
- 冷静期建议。
- 后续动作。

## 10. 演示案例

### 10.1 购物决策案例

用户输入：

```text
我想买一副 1299 元的降噪耳机，最近学习需要安静。
```

系统应完成：

- 追问预算和已有耳机。
- 正方分析学习场景价值。
- 反方分析冲动风险、闲置风险和平替方案。
- RAG 检索历史闲置记录或预算记录。
- 成本工具计算预算占比。
- 法官建议购买、暂缓或拒绝。
- 如果暂缓，加入观察清单并设置提醒。

### 10.2 时间决策案例

用户输入：

```text
我要不要参加这个社团活动？会占用周末两天。
```

系统应完成：

- 追问当前任务和活动收益。
- 正方分析成长价值。
- 反方分析时间成本和延期风险。
- RAG 检索历史任务拖延记录。
- 成本工具计算时间占用。
- 法官给出参加、部分参加或拒绝建议。
- 生成后续提醒或待办。

## 11. MVP 完成定义

MVP 完成必须满足：

- 至少有 2 个完整演示案例。
- 购物和时间两条流程都能跑通。
- 每条流程都经过正方、反方、法官三个 Agent。
- 每条流程至少引用 1 条 RAG 历史记录。
- 每条流程至少调用 1 次 MCP 工具。
- 判决书内容结构完整。
- 前端能展示结果。
- 有测试记录。
