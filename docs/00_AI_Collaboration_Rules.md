# DecisionJury AI 协作开发规矩

## 1. 文档目的

本文件用于约束 DecisionJury 项目开发过程。团队成员都可以使用 AI 辅助开发，但必须保证代码可运行、文档可信、范围可控、协作有序。

本规矩适用于：

- 使用 AI 写代码。
- 使用 AI 改 bug。
- 使用 AI 写文档。
- 使用 AI 生成测试数据。
- 使用 AI 设计接口。
- 使用 AI 生成 Prompt。
- 使用 AI 辅助答辩材料。

## 2. 项目基本约束

- 项目名称：DecisionJury。
- 项目周期：三周。
- 团队人数：5 人。
- 项目范围：购物决策和时间决策。
- 项目形态：Web 应用 + Agent 后端 + RAG + MCP 工具。
- 项目目标：交付一个可演示、可部署、满足课程最低技术要求的多 Agent 冷静决策助手。

## 3. 项目边界

系统只处理低风险日常决策：

- 购物决策：买不买某个商品。
- 时间决策：要不要参加活动、接任务或投入某件事。

系统不处理：

- 医疗决策。
- 法律决策。
- 投资理财决策。
- 借贷决策。
- 就业离职决策。
- 亲密关系决策。
- 任何可能造成重大现实后果的决策。

当用户输入高风险问题时，系统必须拒绝裁决，并说明项目只支持低风险日常决策。

## 4. AI 使用原则

### 4.1 可以做

- 可以让 AI 辅助生成代码初稿。
- 可以让 AI 解释报错。
- 可以让 AI 生成测试用例。
- 可以让 AI 帮助设计 Prompt。
- 可以让 AI 生成文档初稿。
- 可以让 AI 帮助做代码审查。
- 可以让 AI 辅助整理答辩 PPT 内容。

### 4.2 不可以做

- 不允许直接复制 AI 生成的大段代码后不理解就提交。
- 不允许提交没有本地运行过的 AI 代码。
- 不允许让 AI 擅自扩大项目范围。
- 不允许让 AI 编造测试结果。
- 不允许让 AI 编造市场数据、论文引用或接口能力。
- 不允许让 AI 输出“你必须买”“你必须拒绝”等强制性结论。
- 不允许把 API Key、Token、账号密码提交到 GitHub。

## 5. AI 生成内容验收规则

### 5.1 代码验收

AI 生成代码必须检查：

- 是否能本地运行。
- 是否符合项目目录结构。
- 是否使用当前 API 文档中的字段。
- 是否引入新依赖。
- 是否修改了无关文件。
- 是否存在明显重复代码。
- 是否存在不可解释的复杂逻辑。
- 是否泄露密钥或隐私信息。

### 5.2 文档验收

AI 生成文档必须检查：

- 是否符合 MVP 范围。
- 是否出现了暂不做的功能。
- 是否写清楚负责人和验收标准。
- 是否存在高风险决策表述。
- 是否能指导团队成员执行。

### 5.3 Prompt 验收

Prompt 必须检查：

- 是否明确 Agent 角色。
- 是否明确输入字段。
- 是否明确输出格式。
- 是否要求引用 RAG 证据。
- 是否要求使用工具调用结果。
- 是否禁止编造历史记录。
- 是否限制高风险决策。

## 6. GitHub 协作规则

### 6.1 分支规则

项目使用以下分支：

```text
main
dev
feature/agent-core
feature/rag
feature/mcp-tools
feature/frontend
feature/docs-deploy
```

规则：

- 禁止直接向 `main` 提交代码。
- `main` 只保存稳定可演示版本。
- `dev` 用于集成开发版本。
- 每个人从 `dev` 创建自己的功能分支。
- 功能完成后通过 Pull Request 合并到 `dev`。
- 阶段测试通过后，再由负责人把 `dev` 合并到 `main`。

推荐流程：

```text
feature/* -> dev -> main
```

### 6.2 开发前流程

每次开始开发前执行：

```bash
git checkout dev
git pull origin dev
git checkout -b feature/your-task-name
```

### 6.3 提交 PR 规则

PR 标题格式：

```text
feat: implement cost analyzer tool
```

PR 描述必须包含：

```markdown
## 做了什么
- 

## 如何测试
- 

## 影响范围
- 

## 风险
- 

## Checklist
- [ ] 本地已运行
- [ ] 未提交 API Key
- [ ] 接口变更已更新 docs/04_API.md
- [ ] 功能范围符合 docs/01_MVP.md
- [ ] 至少 1 名成员 review
```

## 7. Commit 规则

使用以下格式：

```text
feat: add case creation api
fix: handle empty budget input
docs: update api examples
test: add shopping decision test case
refactor: split agent workflow
chore: update dependencies
```

禁止使用：

```text
update
111
修改
final
```

## 8. 代码质量规则

- 每个模块只做自己的职责。
- 不做与当前任务无关的重构。
- 不随意修改别人负责的文件。
- 公共数据结构必须先写进文档再实现。
- 接口字段修改必须同步更新 `docs/04_API.md`。
- 功能范围修改必须同步更新 `docs/01_MVP.md` 和 `docs/02_SPEC.md`。
- Prompt、工具 schema、接口字段必须保持稳定，不要频繁随意改。

## 9. 数据与隐私规则

- 演示数据使用模拟数据。
- 不使用真实敏感消费记录。
- 不保存身份证、手机号、银行卡、支付账号。
- 不接入真实支付。
- 不接入真实电商账号登录。
- 如果演示使用真实商品链接，必须避免暴露个人账号信息。

## 10. 测试规则

每次合并前至少验证：

- 后端能启动。
- 前端能打开。
- 能创建购物决策案件。
- 能创建时间决策案件。
- 正方 Agent、反方 Agent、法官 Agent 能按顺序输出。
- RAG 能返回历史记录。
- 至少两个 MCP 工具能被调用。
- 能生成最终判决书。

## 11. 每日同步规则

每天每个人同步：

```text
昨天完成：
今天计划：
当前卡点：
需要谁配合：
```

项目负责人每天更新一次里程碑进度。

## 12. Definition of Done

一个功能只有同时满足以下条件，才算完成：

- 代码已提交到对应 feature 分支。
- 本地运行通过。
- 至少有一个测试样例或手动测试记录。
- 接口字段与 API 文档一致。
- 如果涉及 Agent 输出，必须有示例输入和示例输出。
- 如果涉及 RAG，必须能展示检索结果。
- 如果涉及 MCP 工具，必须能展示工具调用日志。
- 已通过至少 1 名组员 review。

## 13. 答辩导向规则

开发时必须保留可展示证据：

- Agent 执行流程截图。
- RAG 检索结果截图。
- MCP 工具调用日志。
- 多轮对话示例。
- 最终判决书示例。
- 测试用例结果。

最终演示必须覆盖：

- 一个购物决策案例。
- 一个时间决策案例。
- 一次 RAG 历史记录引用。
- 一次成本计算工具调用。
- 一次冷静期提醒工具调用。

## 14. 最终交付物

最终提交至少包含：

- 源代码仓库。
- README 运行说明。
- 6 份项目文档。
- 答辩 PPT。
- 演示视频。
- 测试截图或测试记录。
- 可运行的 Demo。
